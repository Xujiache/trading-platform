### 3. 技术决策记录

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 响应格式字段名 | `msg` 而非 `message` | 兼容前端 Art Design Pro 的 HTTP 拦截器 |
| 注册成功状态码 | 200 而非 201 | 前端响应拦截器只处理 code===200 |
| 角色权限存储 | JSON 字段存储权限数组 | 简单灵活，支持动态权限配置 |
| 验证码存储 | Redis (带TTL) | 支持过期自动清理和发送频率限制 |
| 文件存储 | 本地磁盘 | 开发阶段简单可靠，后续可扩展 OSS |
| 用户端语言 | UTS (UniApp x) | 按项目原有框架要求 |

### 5. 经验总结

1. **前后端协议要先对齐**：响应格式（msg vs message）的差异在审查阶段暴露，应在编码前确认
2. **页面跳转链路要完整**：注册页跳转协议页、开屏页跳转 webview 等，必须确保目标页面存在
3. **鉴权要贯穿**：文件上传等接口需在前端显式传递 Authorization header
4. **种子数据要齐全**：开屏广告、内容页等配置项需在初始化脚本中预置
