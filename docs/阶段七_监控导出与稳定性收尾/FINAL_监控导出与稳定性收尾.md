### 4. 技术决策记录

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 审计日志 vs 操作日志 | 独立 audit_logs 表 | 审计日志字段更丰富（含变更前后/耗时/目标对象），与 operation_logs 并行 |
| CSV 导出方式 | HTTP 响应流直出 | 无需磁盘存储，前端 blob 直接下载 |
| 批量操作事务策略 | 逐条事务 | 单条失败不影响其他记录 |
| 审计日志写入方式 | 异步 | 不阻塞请求响应 |
| blob 响应处理 | 拦截器旁路 | 检测 responseType 跳过 JSON 校验 |

### 6. 经验总结

1. **blob 响应兼容**：统一 HTTP 拦截器处理 JSON 响应时，CSV/文件下载需要特殊处理。解决方案是在拦截器中检测 `responseType` 旁路 JSON 校验。

2. **审计日志闭环**：仅定义中间件不够，必须实际挂载到关键业务路由才能产生数据。审计日志覆盖的关键路由包括：
   - 财务管理：入金确认/驳回、出金审核/驳回/打款（8个接口）
   - 风控管理：更新参数、处理预警、手动强平（3个接口）
   - 交易管理：管理员平仓、撤单、改价（3个接口）
   - 用户管理：KYC审核、冻结/解冻、重置密码（3个接口）

3. **文档闭环**：收尾阶段文档更新容易遗漏，应确保 CONSENSUS/DESIGN/TASK/TODO/ACCEPTANCE/FINAL/说明文档全部同步更新。

4. **稳定性增强要点**：
   - 优雅停机应按依赖顺序关闭资源（HTTP→应用服务→数据库→缓存）
   - 行情监控需要冷却期防止频繁重启
   - 全局异常应区分致命错误（端口冲突）和可恢复错误
