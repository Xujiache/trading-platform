### 3. 技术决策记录

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 2FA 实现 | speakeasy + qrcode | 成熟TOTP方案，兼容 Google Authenticator |
| 在线客服通信 | HTTP 轮询 | 项目已有轮询模式，复杂度低，满足需求 |
| 风控预警生成 | 集成在 floatingPnlUpdater | 复用已有定时任务，3秒间隔检查保证金比例 |
| 风控参数校验 | 集成在 tradeEngine | 开仓时校验最大杠杆和单品种最大持仓 |
| 工单状态流转 | pending→processing→resolved→closed | 完整闭环，resolved 由管理员手动标记 |

### 4. 质量评估

- 风控预警生成链路完整：floatingPnlUpdater → checkMarginAlerts → risk_alerts / force_close_records
- 风控参数真正生效：tradeEngine 开仓时校验 max_leverage + max_position_per_symbol
- 强平记录含完整字段：margin_ratio、close_price、realized_pnl
- 公告发布自动通知用户，消息中心点击公告跳详情
- 交易密码/资金密码用户端有入口
- 管理员超管保护完整：不可删除/编辑/禁用
- 工单 resolved 状态有明确操作入口

### 5. 经验总结

- 风控模块需要端到端验证，仅有CRUD不够，需确保"生成→展示→处理"链路完整
- 公告等内容模块需要关注"发布→通知→展示→跳转"全链路
- 密码类接口后端就绪后必须同步前端入口
- 数据库 ENUM 状态在前后端应保持一致
